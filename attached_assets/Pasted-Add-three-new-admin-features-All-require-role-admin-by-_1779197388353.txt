BACKEND: new settings routes

  GET /api/admin/settings/branding
    Returns branding settings. 
    Returns logo/photo as base64 data URL 
    if set, empty string if not.

  PATCH /api/admin/settings/branding
    Accepts multipart/form-data OR JSON.
    For logo and photo uploads: accept 
    base64 string in JSON body 
    (frontend converts file to base64 
    before sending).
    Max image size: 500KB. Reject larger.
    Store in settings table as base64.

  GET /api/admin/settings/contact
    Returns all contact_* settings.

  PATCH /api/admin/settings/contact
    Updates contact settings.
    Photo upload same as logo above.

  GET /api/admin/settings/permissions
    Returns parsed permissions_json.

  PATCH /api/admin/settings/permissions
    Updates permissions_json.
    Only callable by role='admin' — 
    this endpoint is never affected by 
    permission overrides (otherwise admin 
    could lock themselves out).

PUBLIC endpoints (no auth, for frontend use):

  GET /api/public/branding
    Returns: footer_text, footer_url, 
    and logo as base64 data URL.
    Called on app load to hydrate branding.

  GET /api/public/contact
    Returns all contact fields.
    Called when Kontakt view renders.

FEATURE 3: Rechtemanagement view

Create: src/views/admin/PermissionsView.tsx
Sidebar label: "Rechtemanagement"
Icon: ShieldCheck (lucide)
Position: after Kontakt-Daten, 
before KI-Tool

CONCEPT:
A matrix of admin features × roles.
For each feature, the admin can configure 
which roles are allowed to access it.
"Admin (a)" is always allowed everywhere 
and cannot be removed — it is shown as 
a locked checkmark.

FEATURES TO CONFIGURE (featureId: label):
  nutzerverwaltung:    "Nutzerverwaltung"
  analyseprotokoll:    "Analyseprotokoll"
  erscheinungsbild:    "Erscheinungsbild"
  kontakt_daten:       "Kontakt-Daten"
  rechtemanagement:    "Rechtemanagement"
  ki_tool:             "KI-Tool"
  mailserver:          "Mailserver"
  versand_analyse:     "Versand-Analyse"

ROLES:
  admin (a)            — always locked ✓
  user_extended (b)    — toggleable
  user (c)             — toggleable

LAYOUT:
Table with columns: 
  Feature | Admin (a) | Erweiterter User (b) | User (c)

Each cell:
  Admin column: locked checkmark (gray, 
    not clickable), tooltip: 
    "Admins haben immer Zugriff"
  Other columns: toggle switch or checkbox
    Checked = role has access
    Unchecked = role cannot see this item

Below the table:
  "Speichern" button (saves entire matrix)
  
  Info box:
    "Änderungen gelten sofort für alle 
    angemeldeten User. Admins behalten 
    immer Zugriff auf alle Bereiche, 
    unabhängig von dieser Konfiguration."

DEFAULT permissions (applied on first run 
via INSERT OR IGNORE on permissions_json):
  {
    "nutzerverwaltung":  ["admin"],
    "analyseprotokoll":  ["admin"],
    "erscheinungsbild":  ["admin"],
    "kontakt_daten":     ["admin"],
    "rechtemanagement":  ["admin"],
    "ki_tool":           ["admin"],
    "mailserver":        ["admin"],
    "versand_analyse":   ["admin"]
  }

ENFORCEMENT:
In the sidebar and in the route guards, 
replace hardcoded role checks 
(e.g. role === 'admin') with a permission 
check function:

  function canAccess(
    featureId: string, 
    userRole: string,
    permissions: Record<string, string[]>
  ): boolean {
    if (userRole === 'admin') return true
    const allowed = permissions[featureId] ?? ['admin']
    return allowed.includes(userRole)
  }

Load permissions on app start via 
GET /api/admin/settings/permissions 
(requires auth). Store in auth context.

Apply canAccess() to:
  - Sidebar item visibility
  - Route-level guards in each admin view
  - API endpoints (check role against 
    permissions before processing)

IMPORTANT: The /api/admin/settings/permissions 
PATCH endpoint always requires role='admin' 
regardless of the permissions matrix — 
an admin cannot configure themselves out 
of rechtemanagement.

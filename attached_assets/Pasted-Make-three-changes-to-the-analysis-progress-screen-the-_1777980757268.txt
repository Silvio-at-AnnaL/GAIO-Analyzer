CHANGE 4 — Force all progress steps to be 
marked as completed

Currently the steps "Technisches SEO", 
"Schema.org / Strukturierte Daten" and 
"Heading-Struktur" (and sometimes others) are 
skipped in the progress display — they jump from 
"waiting" directly to the next step without ever 
showing as "running" or "done". The results are 
still generated correctly, but the UI skips 
marking them.

Root cause: these steps are fast (no API call 
needed) and complete before the frontend receives 
a progress event for them. The frontend never 
sees them as "running" because they are already 
done when the next event arrives.

Fix this on the BACKEND: before moving to the 
next analysis step, always emit a progress event 
for the current step with status "running", then 
a short pause, then emit "done". This ensures 
the frontend always sees every step transition.

Specifically, in the backend analysis pipeline, 
enforce this pattern for EVERY step:

  # Before running each step:
  emit_progress(step_name, status="running")
  
  # Run the actual analysis:
  result = run_step()
  
  # After completing:
  emit_progress(step_name, status="done")

For the fast synchronous steps (Technical SEO, 
Schema.org, Heading Structure) that complete 
almost instantly, add a minimal artificial delay 
of 300ms between the "running" emit and the 
actual computation — just enough for the frontend 
to receive and render the "running" state before 
"done" arrives:

  emit_progress("Technisches SEO", "running")
  await asyncio.sleep(0.3)
  tech_result = analyze_technical_seo(...)
  emit_progress("Technisches SEO", "done")

Apply this to every single step in the pipeline 
without exception:
  1. Crawling Website
  2. Technisches SEO
  3. Schema.org / Strukturierte Daten
  4. Heading-Struktur
  5. Inhaltliche Relevanz (KI-Analyse)
  6. FAQ-Qualität
  7. LLM-Auffindbarkeit
  8. Wettbewerbsvergleich
  9. Empfehlungen generieren

The 300ms delay only applies to the three fast 
synchronous steps (2, 3, 4). All other steps 
already take longer than 300ms naturally and 
need no artificial delay.

Also fix the step label "FAQ-Qualitaet" (with 
umlaut missing): rename it to "FAQ-Qualität" 
in both the frontend step list and any backend 
progress event names.
